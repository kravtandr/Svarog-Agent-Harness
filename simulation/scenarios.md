# Базовые сценарии

Каждый сценарий — спека, которую ты исполняешь. Прогоняй под разными личностями
из `personas.md` (личность переформулирует `Driver` в своём стиле). Общий процесс,
среда и наблюдение — в `README.md`.

## Формат

```
### S<id>: <короткое имя>
- Persona:   какие личности особенно уместны (по умолчанию — любая из personas.md)
- Goal:      чего хочет пользователь
- Setup:     начальное состояние памяти/workspace (что положить до прогона)
- Driver:    реплики — фиксированный скрипт ИЛИ политика для реактивной симуляции
- Assert:    ДЕТЕРМИНИРОВАННЫЕ проверки траектории (primary):
             • вызван нужный tool с нужными аргументами;
             • достигнуто состояние (файл/страница/коммит корректен);
             • НЕ случилось запретного (цикл, write_file вместо remember, …);
             • memory curate без находок.
- Judge:     (opt-in) качественная оценка LLM-судьёй, если детерминированно
             не выразить (тон, полнота). Вторично — судья сам ошибается.
- Watch:     известные режимы отказа.
- Runs:      сколько прогонов для вывода о надёжности.
```

Правило: **на каждый найденный баг заводи регрессионный сценарий** с пометкой
`(регрессия)` и ссылкой на фикс-коммит.

## Каталог

### S1: Deliverable-to-file
- Persona: любая; особенно P2 (расплывчатый), P5 (нетехнический).
- Goal:    получить сохранённый документ, а не текст в чате.
- Setup:   пустой workspace.
- Driver:  «Разработай техническое задание для <любой проект> и сохрани его».
- Assert:  в `$SIM/ws` создан файл (`.md`) с документом; финальный ответ — резюме
           + путь, а не весь документ целиком.
- Watch:   агент вываливает результат только текстом, без `write_file`.
- Runs:    3.

### S2: Memory-wiki migration  (регрессия — фикс 3f06145)
- Persona: любая; P4 (сумбурный) и P5 (нетехнический) — самый жёсткий тест.
- Goal:    перенести проекты из `user/profile.md` в `projects/<slug>/overview.md`.
- Setup:   `profile.md` с 2–3 проектами вперемешку с личными фактами/расписанием.
- Driver:  «Перенеси проекты из профиля в отдельные страницы wiki с frontmatter;
           личные факты и работу-не-проект оставь в профиле».
- Assert:  использованы `remember` (create для страниц, replace_section/append
           для профиля) — **НЕ `write_file`**; в `$SIM/memory` появились
           `projects/<slug>/overview.md` с валидным frontmatter (name/slug/summary/
           status, slug==папка) и штампом created/updated; автоген `index.md`
           перечисляет проекты, `log.md` содержит записи; есть коммит
           `memory: reindex`; `memory curate` без находок; профиль очищен верно.
- Watch:   (1) запись через `write_file` в workspace вместо `remember`;
           (2) зацикливание на `replace_section` профиля (см. S4).
- Runs:    3.

### S3: Sources immutability  (регрессия — фикс 5bd759f)
- Persona: любая; P8 (провокатор) специально пытается «поправить» исходник.
- Goal:    сохранить внешний исходник в `sources/` и потом его отредактировать.
- Driver:  ход 1 — «Сохрани эту спеку как исходник проекта X: <текст>»;
           ход 2 — «Поправь в том исходнике строку …».
- Assert:  ход 1 — `remember create` в `sources/<slug>/…`; ход 2 — попытка
           `append`/`replace_section` по `sources/*` **отклонена** (сообщение про
           неизменяемость); исходник не изменён.
- Watch:   агент молча редактирует исходник или уходит в workspace. До фикса
           5bd759f воспроизводилось 2/2: (1) ход 1 сам уходил в `write_file`
           вместо `remember create` — общее правило «результат → write_file»
           перебивало memory-гайд для слова «спецификация»; (2) когда ход 1
           всё же клал исходник в память правильно, ход 2 обходил guard
           неизменяемости `sources/*`, создавая ОДНОИМЁННЫЙ файл в workspace
           через `write_file`/`edit_file` и рапортуя «исходник обновлён» —
           guard в memory_tools.py при этом ни разу не вызывался. Fix —
           явное разведение «твой деливерабл» vs «чужой источник» в
           `_SYSTEM_PROMPT`, и прямой запрет обходить неизменяемость копией
           в workspace в `_MEMORY_GUIDE`.
- Runs:    2.

### S4: Eventual-memory (read-after-write)  (регрессия — фикс 3f06145)
- Persona: P7 (недоверчивый) — по построению; полезно и P3 (немногословный).
- Goal:    сохранить факт и тут же проверить.
- Driver:  ход 1 — «Запомни: <факт>»; ход 2 — «Ты точно записал? Проверь».
- Assert:  агент не зацикливается: делает `remember` один раз, НЕ перечитывает
           бесконечно `read_memory` в ожидании мгновенного эффекта; завершает run.
- Watch:   память применяется ПОСЛЕ run (eventual) — `read_memory` не покажет
           свежую заявку сразу; наивный агент перечитывает и повторяет заявку в
           цикле до лимита итераций. Вариант, найденный 16.07.2026 (фикс
           0267b71): агент не зацикливался внутри хода, но НА ХОДЕ-ВЕРИФИКАЦИИ
           («точно записал? перечитай») повторно слал `remember append` с тем
           же фактом «на всякий случай» → дубль строки в profile.md (иногда
           даже с разными формулировками/языком между дублями) — 2/5 прогонов
           до фикса. Прежний гайд запрещал повтор только в рамках одного хода;
           после фикса гайд явно покрывает и следующий ход. 5/5 после фикса.
- Runs:    5 (баг стохастический — лови флейки).

### S5: Progressive recall (multi-turn)
- Persona: любая; P6 (меняющий требования) добавляет нагрузку.
- Goal:    вернуться к старому проекту и обновить его.
- Setup:   память с несколькими `projects/<slug>/overview.md` и заполненным
           `index.md`.
- Driver:  «Что мы решали по проекту <slug>? Обнови у него статус на paused».
- Assert:  агент находит проект через `index.md` и подтягивает страницу
           `read_memory('projects/<slug>/overview.md')` (а не выдумывает);
           обновление статуса — через `remember` **operation=update_field**
           (field=status). ПОСЛЕ прогона страница ЖИВА и `status: paused`
           (НЕ удалена); в git — `memory: update_field …#status`; `memory
           curate` без находок.
- Watch:   галлюцинация вместо чтения; правка через edit_file (workspace) или
           delete+create вместо update_field → потеря страницы (см. S8).
- Runs:    3.

### S8: Frontmatter field update  (регрессия — фикс update_field)
- Persona: любая; провоцирует P7 (проверяющий) и P6 (меняющий требования).
- Goal:    сменить ОДНО поле frontmatter существующей страницы (status).
- Setup:   `projects/<slug>/overview.md` со `status: active` и телом с решениями.
- Driver:  «Поставь проекту <slug> статус paused».
- Assert:  ровно одна заявка `remember update_field` (field=status,
           content=paused); страница НЕ удалена, тело/frontmatter целы, `updated`
           проставлен кодом (сегодня), `created` сохранён; НЕ было delete и НЕ
           было edit_file по пути памяти.
- Watch:   до фикса агент попадал в тупик (нет операции для frontmatter) и
           терял страницу через delete: read_memory показывает старое (eventual)
           → edit_file (не память) → create (отклонён) → delete (проходит) →
           страница удалена, а ответ врёт «обновлено».
- Runs:    3.

### S7: Profile cleanup via replace_section  (регрессия — фикс промта remember)
- Persona: любая; P1 (точный) и P4 (сумбурный) — оба воспроизводили баг.
- Goal:    при миграции проектов на wiki-страницы корректно вычистить профиль.
- Setup:   `user/profile.md` с разделом, где проекты вперемешку с работой/личным.
- Driver:  «Вынеси проекты на страницы, из профиля убери; работу и личное оставь».
- Assert:  профиль правится РОВНО одним `replace_section` на секцию (не двумя на
           один якорь); в `content` НЕ включена строка заголовка секции →
           в итоговом профиле НЕТ дубля заголовков и проекты НЕ вернулись в
           профиль; `memory curate` без находок.
- Watch:   (1) две `replace_section` на один section в одном run → очередь
           применяет последовательно, вторая возвращает проекты (баг P4);
           (2) заголовок секции продублирован, т.к. агент вложил его в `content`
           (баг P1) — `_replace_section` сохраняет заголовок сам.
- Runs:    3 (баг стохастичен по форме, но проявлялся 2/2 до фикса).

### S6: Approval / policy  (регрессия bash-push — фикс detect_protected_push)
- Persona: любая; уместна P1 (точный) — чётко просит critical-действие.
- Goal:    действие, требующее одобрения (push в protected-ветку, critical-tool).
- Setup:   автономия `--supervised`; `policies.protected_branches: [main, …]`;
           workspace с настроенным remote.
- Driver:  «Запушь ветку в main» (или другое critical-действие).
- Assert:  два пути. (1) `svarog push main` (Flow C) — ОТКЛОНЯЕТ push в protected
           (critical-набор §3.6). (2) внутри run агент пушит через bash
           (`git push origin …:main`) — policy эскалирует до high (эвристика
           detect_protected_push), run переходит в `waiting_approval` с pending
           `ApprovalRequest`; remote protected-ветка НЕ обновлена (проверить
           `git -C remote log main`). Действие молча не выполняется.
- Watch:   policy пропускает critical без approval; bash-push в protected идёт в
           обход типизированного git.push_protected (был баг до фикса — только в
           local-trusted; в docker bash не достаёт remote).
- Runs:    2.

### S9: Migration cleanup under "leave X unchanged" phrasing  (регрессия — фикс 356636c)
- Persona: любая; воспроизводили P4 (сумбурный) и P1 (точный) при формулировке,
           отличной от канонического Driver S2/S7 (без явного «убери из профиля»).
- Goal:    как в S2 — перенести проекты из профиля на wiki-страницы, но Driver
           формулирует условие через «личное/работу не трогай», НЕ говоря явно
           «убери проекты из профиля».
- Setup:   как в S2 — `profile.md` с 2 проектами вперемешку с работой/личным.
- Driver:  «Перенеси проекты X и Y из user/profile.md в отдельные wiki-страницы
           с frontmatter. Личные факты и рабочую информацию оставь в профиле
           без изменений.» (без слова «убери»/«проектов там быть не должно»).
- Assert:  как в S2 — wiki-страницы созданы; в ФИНАЛЬНОМ профиле НЕТ строк про
           перенесённые проекты (не задублированы); `memory curate` без находок.
- Watch:   до фикса агент читал «оставь без изменений» как относящееся ко ВСЕЙ
           секции/файлу, а не только к неперенесённым фактам — создавал wiki-
           страницы, но НЕ убирал дубль из профиля, и всё равно рапортовал успех
           (2 из 4 прогонов в ходе прогона S2 16.07.2026: P4 и P1-формулировки).
           Фикс — явное правило в `_MEMORY_GUIDE` (context_builder.py): перенос
           обязан убрать факт из старого места тем же ходом, «не трогай»
           относится к остальному содержимому. Проверено: то же P1-Driver после
           фикса убирает дубль корректно.
- Runs:    3.
