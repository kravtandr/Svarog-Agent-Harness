# Svarog — Agent-Based User Simulation

Инструкции для **внешнего кодинг-агента** (Claude Code, Codex, OpenCode — любого),
который тестирует агент Svarog, симулируя пользователя. Ты прогоняешь **реального
агента Svarog с реальными вызовами LLM** в изолированной среде, ведёшь его по
сценариям от лица разных личностей, наблюдаешь полную траекторию, проверяешь
поведение и чинишь регрессии. Ты можешь добавлять, менять и динамически ветвить
сценарии.

Это систематизация ручной отладки: однажды человек так нашёл 2 бага промта за 3
прогона миграции памяти. Твоя задача — делать это методично и широко.

## Файлы папки

| Файл | Что содержит |
|---|---|
| `README.md` (этот) | как всё устроено: среда, запуск, наблюдение, цикл, безопасность |
| `scenarios.md` | базовые сценарии (цель + setup + детерминированные проверки) |
| `personas.md` | личности пользователя (стиль формулировок) для стресс-теста |

Матрица тестирования = **сценарий × личность**: одну и ту же цель личность
переформулирует в своём стиле (точно / расплывчато / немногословно / …). Так
проверяется устойчивость агента к тому, КАК человек говорит, а не только К чему.

---

## 1. Ментальная модель

- **SUT (система под тестом)** — агент Svarog: autonomous runtime (loop + tools +
  memory + sandbox + policy). По умолчанию ты его гоняешь и наблюдаешь, а не
  переписываешь.
- **Ты играешь три роли:** *User Simulator* (реплики от лица личности из
  `personas.md`), *Scenario driver* (идёшь по сценарию из `scenarios.md`,
  ветвишься), *Trajectory Evaluator* (проверяешь траекторию → PASS / FAIL / FLAKY).
- **Реальный LLM = стохастика.** Один прогон ничего не доказывает про надёжность:
  для проверок устойчивости гоняй **N раз** (3–5) и смотри долю успеха.
- **Изоляция обязательна.** Никогда не гоняй против реального `agent-home/memory`
  пользователя или его рабочего репозитория. Только одноразовые temp-каталоги.

## 2. Развёртывание изолированной среды (проверенный рецепт)

Ключевые факты, без которых будет молча не работать:
- **Память должна быть собственным git-репо.** Иначе memory-writer резолвится в
  родительский репозиторий и молча ничего не коммитит. Всегда `git init` памяти.
- **Workspace — отдельный одноразовый git-репо** с начальным коммитом (Flow C
  создаёт task-ветку и коммитит туда).
- **Секрет провайдера** (`PROVIDER_API_KEY` или как назван в конфиге) доступен в
  `~/.svarog/secrets.json` или по `secrets.path`; `HOME` при запуске указывает
  туда, где лежит store.
- **sandbox: `local-trusted`** — для задач про память/тексты (быстро, без docker);
  `docker` — только если сценарий реально исполняет недоверенный код.

```bash
REPO=/path/to/Svarog            # корень репозитория svarog_harness
SIM=$(mktemp -d)                # песочница симуляции
mkdir -p "$SIM/state"

# 1) Память как собственный git-репо
mkdir -p "$SIM/memory/user" "$SIM/memory/projects" "$SIM/memory/decisions"
printf '# Профиль пользователя\n' > "$SIM/memory/user/profile.md"
git -C "$SIM/memory" init -q
git -C "$SIM/memory" add -A
git -C "$SIM/memory" -c user.name=Sim -c user.email=sim@localhost commit -qm baseline

# 2) Одноразовый workspace
mkdir -p "$SIM/ws"
git -C "$SIM/ws" init -q
echo scratch > "$SIM/ws/README.md"
git -C "$SIM/ws" add -A
git -C "$SIM/ws" -c user.name=Sim -c user.email=sim@localhost commit -qm init

# 3) Конфиг (подставь свой провайдер/модель/секрет)
cat > "$SIM/ws/svarog.yaml" <<YAML
models:
  default: local
  providers:
    local:
      type: openai-compatible
      base_url: https://openrouter.ai/api/v1
      model: openai/gpt-oss-120b
      api_key_ref: PROVIDER_API_KEY
runtime:
  autonomy: yolo
sandbox:
  type: local-trusted
memory:
  path: $SIM/memory
storage:
  db_path: $SIM/state/svarog.db
YAML
```

Альтернатива: `svarog init <dir>` создаёт готовый agent-home и **правильно
git-инициализирует память**; конфиг всё равно надо отредактировать под провайдера.

## 3. Как прогнать сценарий

```bash
cd "$SIM/ws"
HOME="$HOME" uv run --project "$REPO" svarog run --yolo "<реплика личности>"
```

- **Одноходовые сценарии** (большинство) — `svarog run --yolo "<task>"`. Прогон
  печатает траекторию в stdout: `→ tool {...}` на каждый вызов, финальный ответ,
  `память обновлена (<sha>)`, строку `completed | run <id> | N итераций | tokens`.
- **Долгие прогоны** (реальный LLM) — запускай в фоне и следи за output-файлом;
  агент может зациклиться — тогда останавливай по таймауту (это тоже находка).
- **Многоходовые (реактивные)** — `svarog chat` (каждая строка stdin = run в общей
  session; пустая строка/`/quit` — выход). Веди диалог построчно: прочитал ответ
  Svarog → сформулировал следующую реплику по личности/цели → подал. Нужен
  интерактивный ввод (pty/expect); если сторона пользователя предопределена —
  можно подать заготовленную последовательность строк.

Автономия: `--yolo` (без approvals) по умолчанию; `--supervised`/`--auto` — если
сценарий проверяет approval/policy.

## 4. Что наблюдать (сбор траектории)

Проверяй **не только финальный текст** — весь путь:

| Источник | Как смотреть | Что даёт |
|---|---|---|
| stdout прогона | `→ tool {...}`, финальный ответ, `completed \| … итераций` | какие tools вызваны, сколько итераций, зациклился ли |
| git-лог памяти | `git -C "$SIM/memory" log --oneline` | применились ли заявки; есть ли `memory: reindex` |
| index.md / log.md | `cat "$SIM/memory/index.md"`, `.../log.md` | сработал ли автоген каталога/журнала |
| страницы проектов | `cat "$SIM/memory/projects/<slug>/overview.md"` | frontmatter, штамп дат |
| lint памяти | `HOME="$HOME" uv run --project "$REPO" svarog memory curate` | orphan/invalid/stale/empty |
| трейс (авторитетный) | `svarog traces list` / `svarog traces show <id>`; либо SQLite `$SIM/state/svarog.db` (`messages`, `tool_calls`, `runs`) | полная последовательность сообщений и вызовов |
| workspace | `git -C "$SIM/ws" log --oneline`, дерево файлов | что агент записал в workspace |

Различай назначения: **`write_file` → workspace**, **`remember` → память**. Файл в
`$SIM/ws/projects/...` вместо `$SIM/memory/projects/...` — баг маршрутизации.

## 5. Цикл, который ты исполняешь (динамическая разведка)

```
выбери сценарий (scenarios.md) × личность (personas.md)
  → разверни изолированную среду (раздел 2)
  → положи Setup сценария, переформулируй Driver в стиле личности
  → прогони SUT (раздел 3), собери траекторию (раздел 4)
  → проверь Assert сценария → PASS / FAIL / FLAKY
      FAIL  → диагностируй корень (промт? код? инфра?), предложи/применяй фикс,
              перепрогони до зелёного, заведи регрессионный сценарий
      FLAKY → гоняй N раз, оцени долю успеха, ищи стохастический триггер
  → зафиксируй результат (сценарий, личность, прогоны, вывод, ссылка на трейс)
  → ВЕТВИСЬ: всплыло новое поведение — породи вариант (другая личность/крайний
            ввод/соседний tool), добавь в каталог
```

Идя динамически, реагируй на фактический ответ агента: задал уточняющий вопрос —
отвечай по личности; свернул не туда — либо подыграй (проверить восстановление),
либо зафиксируй отклонение.

Бюджет: реальный LLM платный. Предпочитай локальную/auxiliary модель, ограничивай
число прогонов.

---

## 6. Защитные правила (обязательно)

- Работай ТОЛЬКО во временных каталогах (`mktemp -d`). Никогда не трогай реальный
  `agent-home/memory` и рабочий репозиторий пользователя.
- Не коммить песочницу симуляции в репозиторий Svarog.
- Реальные LLM-вызовы платные — батчируй, ставь лимиты, чисти за собой.
- Если для фикса нужны правки кода/промта Svarog — делай их на ветке, прогоняй
  `uv run pytest -q`, и только после зелёного перепрогони сценарий на реальном
  агенте и считай фикс подтверждённым.

---

## 7. Как добавлять и менять

- Новый сценарий — блок по шаблону в `scenarios.md`. Любой найденный баг →
  регрессионный сценарий с пометкой `(регрессия)` и ссылкой на фикс-коммит.
- Новая личность — блок в `personas.md`.
- Варианты порождай осознанно: меняй ОДНУ ось за раз (личность / крайний ввод /
  начальное состояние / автономия / соседний tool) — так отказ локализуется.
