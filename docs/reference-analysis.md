# Анализ референсных проектов: hermes-agent, HKUDS/OpenHarness и HKUDS/nanobot

Первые два репозитория склонированы в `/home/kravtandr/reference/` (shallow clone, 2026-07-08). Оба под **MIT** — код можно переиспользовать напрямую с сохранением копирайта. Третий, HKUDS/nanobot (§3), разобран точечно в рамках спеки блока A «контекст и надёжность хода» (2026-07-20, `docs/superpowers/specs/2026-07-20-context-turn-reliability-design.md`), без полного клонирования — тоже под **MIT**, совместимой с Apache-2.0 проекта.

## 1. NousResearch/hermes-agent

Зрелый production-агент (~сотни модулей): TUI, gateway на 6 платформ (Telegram, Discord, Slack, WhatsApp, Signal, CLI), 6 execution-окружений, curator, компакция контекста, cron-планировщик, subagents, batch trajectory generation. Python, монолитный процесс.

### 1.1. Curator — главное, что нас интересует

`agent/curator.py` (~2000 строк) + `tools/skill_usage.py`. Устройство подтверждает нашу двухслойную модель и добавляет проверенные практикой детали:

**Слой 1 — детерминированный** (`apply_automatic_transitions`): lifecycle `active → stale → archived` по timestamp последней активности (stale после 30 дней, archive после 90, конфигурируемо). Применяется автоматически, без LLM и без approval.

**Слой 2 — LLM-консолидация** (`run_curator_review`): форк агента на **auxiliary-модели** (дешевой, отдельной от основной сессии — не портит prompt cache) может pin/archive/consolidate/patch скиллы. **Выключен по умолчанию** (opt-in) — консолидация «опинионейтед» и стоит денег.

**Инварианты, которые стоит перенять дословно:**

* curator трогает **только agent-created** скиллы (provenance-маркировка при создании через skill_manage); bundled и вручную написанные — вне зоны действия;
* **никогда не удаляет** — только архивирует в `.archive/` (обратимо);
* `pinned` — флаг, выводящий скилл из-под любых авто-переходов;
* скиллы, на которые ссылается **любой cron job** (включая paused), считаются используемыми — иначе редко срабатывающая автоматизация теряет свой скилл;
* новый скилл получает якорь `created_at`, чтобы не заархивироваться сразу;
* телеметрия использования — в **sidecar-файле** (`.usage.json`), не во frontmatter SKILL.md: операционные данные не загрязняют человекочитаемый контент и не создают конфликтов;
* запуск — **inactivity-triggered** (агент idle + прошло `interval_hours`), без отдельного cron-демона;
* каждый прогон пишет markdown-отчет.

### 1.2. Approval (`tools/approval.py`)

* `DANGEROUS_PATTERNS` — готовый выверенный набор паттернов опасных bash-команд → прямой донор для наших bash-эвристик (слой 2 ADR-0002);
* **YOLO-режим замораживается при старте процесса**: env-переменная читается один раз при импорте, иначе скилл мог бы выставить её изнутри и снять все проверки — классическая prompt-injection-эскалация. Принцип обязателен и нам;
* smart approval: aux-LLM авто-подтверждает низкорисковые команды;
* постоянный allowlist в конфиге.

### 1.3. Execution environments (`tools/environments/`)

Единая модель «spawn-per-call»: каждая команда — свежий `bash -c`, снапшот сессии (env, функции, aliases) снимается при init и восстанавливается перед командой, cwd переживает вызовы. Backends: `local`, `docker`, `ssh`, `singularity`, `modal`, `daytona` за одним базовым классом. Это готовая реализация нашего интерфейса `SandboxBackend` — минимум `docker.py` + `local.py` стоит адаптировать, а не писать с нуля.

### 1.4. Компакция контекста (`agent/context_compressor.py`)

Aux-модель суммирует середину разговора, защищая голову и хвост (по токен-бюджету, не по числу сообщений); структурированный шаблон summary (Resolved/Pending); дешевый pre-pass обрезки tool outputs перед LLM; итеративное обновление summary между компакциями; секции «historical/reference-only», чтобы старое summary не читалось как активные инструкции. Готовый дизайн для нашей пост-MVP компакции.

### 1.5. Прочее ценное

* **Auxiliary model** как отдельная сущность конфигурации: дешевая модель для компакции, curator, smart approval, суммаризации сессий;
* FTS5-поиск по прошлым сессиям + LLM-суммаризация для cross-session recall;
* «nudges» — периодические напоминания агенту сохранять знания в память/скиллы;
* `MemoryManager` — provider-модель памяти (prefetch перед ходом / sync после хода);
* checkpoint_manager, iteration_budget, delegate_tool (изолированные subagents);
* code execution RPC: агент пишет Python-скрипт, вызывающий tools, — многошаговый pipeline за один ход без расхода контекста;
* совместимость SKILL.md с открытым стандартом [agentskills.io](https://agentskills.io).

## 2. HKUDS/OpenHarness

Молодой проект (v0.1.x): «oh» — lightweight harness (engine, 43 tools, skills, memory, permissions, MCP, swarm/coordinator) + «ohmo» — персональный агент поверх него. Python ядро + React Ink TUI. Работает поверх подписок Claude Code/Codex (bridge), а не только напрямую с API.

Ценное:

* **`skills/_frontmatter.py` + `loader.py`** — маленький чистый парсер SKILL.md, совместимый с тем же стандартом; `SkillDefinition` с полезными полями: `user_invocable`, `disable_model_invocation`, `aliases`, `argument_hint` (скилл = одновременно slash-команда для человека);
* **permission modes**: `default / plan / full_auto` — подтверждает наш дизайн профилей автономии; **plan mode** (агент строит план без исполнения) — режим, которого у нас нет;
* **memdir**: project-scoped память с entrypoint `MEMORY.md`, ограниченным по байтам/строкам, и policy-строками — деталь «лимит на entrypoint» стоит взять;
* `query_engine` (~300 строк) — пример компактного agent loop: streaming, retry с backoff, параллельные tool calls, cost tracking.

Осознанно **не** берем: авторизацию поверх чужих подписок (bridge к Claude Code/Codex) — у нас model-agnostic API-подход; React Ink TUI (у нас Typer/Rich, TUI — не MVP).

## 3. HKUDS/nanobot

Лицензия **MIT**. В отличие от §1–2 (широкий разбор всего проекта), nanobot
разбирали прицельно — под конкретную задачу «блок A: контекст и надёжность
хода» (`docs/superpowers/specs/2026-07-20-context-turn-reliability-design.md`,
план `docs/superpowers/plans/2026-07-20-context-turn-reliability.md`), а не
целиком. HKUDS/OpenHarness (§2) отдельно вендорит слой каналов/WebUI nanobot
(`hkuds-openharness/scripts/sync_nanobot_channels.sh`) — это подтверждает,
что nanobot существует как отдельный апстрим, не то же самое, что OpenHarness.

### 3.1. Что перенесено (блок A, реализовано — ADR-0015 §6)

* **Инвариант истории перед вызовом модели** (`nanobot/agent/context_governance.py`) —
  у nanobot это активный ремонт: перед каждым `complete()` он удаляет orphan
  tool-результаты, дописывает заглушки для незакрытых `tool_call`, вычищает
  malformed-вызовы. У Svarog write-ahead-checkpoint (`LoopState.pending_tool_calls`
  в checkpoint до исполнения, ADR-0005) делает orphan-состояния структурно
  невозможными — перенесена не логика ремонта, а сама идея «проверять форму
  истории перед каждым вызовом модели», реализованная как read-only assert
  (`runtime/history_invariant.py`), а не ремонт. → ADR-0015 §6.1.
* **Actionable-текст маркера компакции** (`context_governance.py:428`,
  `_tool_result_compaction_message`) — приём «компакция как обучающий сигнал
  модели, а не молчаливое усечение» перенесён на формулировку маркера
  микрокомпакции Svarog (ADR-0015 §1.4, уже существовавшей до блока A). →
  ADR-0015 §6.2.
* **Стабильный порядок tool definitions** — идея фиксировать порядок схем
  tools отдельно от алфавитной сортировки ради prefix-cache провайдера,
  вынося изменчивые/поздно-добавленные схемы в конец. → ADR-0015 §6.3.
* **Учёт `cached_tokens`** — сама механика разбора usage трёх диалектов
  openai-compatible API не заимствована из кода nanobot напрямую (это факт
  об API провайдеров, не о nanobot), но именно разбор nanobot подсветил, что
  без метрики попаданий в prompt-cache стабилизацию префикса (пункт выше)
  нечем измерить постфактум. → ADR-0015 §6.3.
* **Ремонт формы tool-вызова** (двойная JSON-сериализация, лишний конверт
  `{"arguments": {...}}`) — механика распознавания искажённых аргументов
  модели. У Svarog реализована как отдельный слой над `ToolCallRequest.parse_arguments`
  в `tools/registry.py` (реестр tools, не llm-слой), не заменяя строгий
  разбор, а достраивая его. → ADR-0015 §6.4.

Идея «мерить, где встал ход» (тайминги фаз, ADR-0015 §6.5) заимствована у
nanobot без механизма: у него ход — конечный автомат с явной таблицей
переходов (`nanobot/loop.py:102,256`; отсутствие перехода — `RuntimeError`),
оправданный множеством входов в ход (слэш-команды, restore checkpoint'а,
автокомпакция, mid-turn инъекции субагентов). У Svarog вход в ход один,
управление — линейный `while`, поэтому вместо FSM — контекст-менеджер
`PhaseTimer`, накапливающий длительности внутри уже существующих участков
цикла.

### 3.3. Блок E: что из него выжило проверку

Блок E значился как три «мелкие hardening/UX» правки. Проверка против кода
оставила одну — две другие в архитектуре Svarog не к чему приложить.

**Перенесено: подсказки для жёстких границ.** Отказ, повтор которого
заведомо бесполезен, объясняет модели, что граница жёсткая и что делать
вместо повтора. Класс границы определяется типизированно в момент отказа,
текст подставляется при рендере для модели и не попадает в trace. Детали и
мотивация — ADR-0002, раздел «Подсказки для жёстких границ».

**Отвергнуто: tmpfs-маскировка agent-home.** У nanobot приём нужен, потому
что их bwrap-песочница ro-биндит корень хоста и каталог с конфигом и ключами
виден изнутри; tmpfs его прячет. Docker-sandbox Svarog монтирует только
workspace, skills и (для внешнего агента) каталог состояния — agent-home не
монтируется вообще, прятать нечего. Остаточный случай `local-trusted`, где
bash работает на хосте и путями не заперт в принципе, — осознанный trade-off
режима (ADR-0015 §0.3); закрыть его можно только новым sandbox-бэкендом с
namespaces, что несопоставимо по объёму с «мелкой правкой». Вместо механизма
инвариант закреплён тестом
`tests/test_sandbox.py::test_docker_run_args_never_mount_agent_home`.

**Отвергнуто: Runtime Context как удаляемый блок метаданных.** У nanobot
механизм защищает от утечки метаданных, дописываемых к сообщению
пользователя. В Svarog к `user`-сообщению не дописывается ничего
(`runtime/context_builder.py` кладёт голый текст задачи, nudges — отдельные
сообщения). Утечки, которую надо предотвращать, нет; механизм без
потребителя не заводится (YAGNI).

### 3.4. Блок B: автопродолжение долгих run'ов

**Перенесено: автопродолжение хода.** При достижении порога refuel run
пересобирает контекст из `task_state.md` и продолжает работу сам, не требуя
`svarog resume`. Потолок — `runtime.max_refuel_rounds` (по умолчанию 12);
`max_iterations` при этом стал лимитом сегмента между refuel'ами, а общими
стоп-кранами — потолок раундов и бюджеты токенов и стоимости. Детали —
ADR-0005, раздел «Автопродолжение после refuel».

**Важная находка по ходу работ:** авто-продолжение в Svarog уже частично
существовало — refuel-supervisor `GatewayService` (`supervisor.auto_resume_refuel`,
включён по умолчанию) поднимал refuel-suspended run'ы в режимах `serve` и
`telegram`. Ни разбор nanobot, ни спек этого не заметили; утверждение «продолжить
run может только человек» было верным лишь для локального CLI. Механизмы
разведены конфигом: при включённом автопродолжении супервизор refuel-остановки
не трогает, потому что теперь они означают исчерпанный потолок — намеренную
остановку.

**Отвергнуто: идемпотентный restore checkpoint.** Суффиксный матчинг nanobot
решает задачу слияния checkpoint'а с уже существующей историей хода. У Svarog
слияния нет: `orchestrator.py` поднимает `LoopState.from_dict(raw_state)` из
блоба checkpoint'а как единственный источник, `resume()` передаёт его в цикл
без примешивания чего-либо из trace. Дублировать нечего. Вместо механизма
инвариант закреплён тестами
`tests/test_resume.py::test_resume_history_comes_only_from_checkpoint` и
`::test_resume_reexecutes_write_ahead_calls_at_least_once`.

Настоящая граница идемпотентности у Svarog другая — at-least-once для
write-ahead вызовов (ADR-0005): между исполнением tool и записью следующего
checkpoint вызов может повториться. Это осознанное свойство, nanobot его тоже
не решает, и в объём блока B оно не входило.

### 3.5. Блок D: планировщик

**Перенесено: ядро расписания и системные джобы.** Появился источник запуска
run'ов, отличный от человека: отдельный демон `svarog scheduler`, таблица
джоб в SQLite, атомарный захват через compare-and-set. Детали — ADR-0019.

Из приёмов nanobot заимствован **defer-until-idle**: джоба, чей workspace
занят интерактивной работой, пропускает тик, а не падает. Но опирается он на
уже существующий per-workspace lease (ADR-0015 §0.5), а не на новый механизм.

**Не перенесено:** heartbeat-файл как источник задач и durable-очередь внешних
триггеров — остались вне объёма цикла; полный cron-синтаксис — потребовал бы
внешней зависимости.

**Осознанно не перенесено: свободный агентский доступ к созданию джоб.** У
nanobot агент заводит cron-джобы инструментом без гейта. Для Svarog это
неприемлемо: модель безопасности замораживает права при старте run'а, а джоба
переживает run и исполняется позже с конфигом на момент срабатывания. Поэтому
`schedule.create` внесён в неотключаемый critical-набор — approval требуется
в любом режиме автономии, включая yolo.

Попутно закрыта заглушка в `skills/curator/pruning.py`: скиллы, используемые
включёнными джобами, больше не архивируются по сроку неактивности. Связь
определяется по телеметрии `SkillLoad`, а не по тексту задачи джобы.

### 3.6. Что отвергнуто (с причинами)

* **Каналы/WebUI целиком** — у nanobot есть многоканальный gateway
  (Telegram/Discord/Slack/WebUI и т.п.; частично уже вендорится в
  HKUDS/OpenHarness, см. `sync_nanobot_channels.sh` выше). Svarog не берёт
  этот слой: своя модель интерфейсов (CLI/gateway, ТЗ §10) архитектурно
  другая, и канал доставки — не про надёжность хода;
* **Инструмент `my`** (самоуправление правами/конфигом изнутри агента) —
  противоречит заморозке autonomy/policy на старте run (ADR-0010; тот же
  принцип — «YOLO-freeze» — уже отмечен для hermes-agent, §1.2): инструмент,
  дающий агенту менять собственные полномочия во время исполнения, открывает
  ровно ту эскалацию через prompt injection, которую заморозка призвана
  закрыть;
* **bwrap внутри Docker** — доп. слой изоляции (`bubblewrap`) поверх
  основного рантайма nanobot. Внутри Docker-контейнера Svarog это требует
  выдать контейнеру `CAP_SYS_ADMIN`, что размывает изоляцию контейнера
  сильнее, чем добавляет: при уже включённом docker-сендбоксе (ADR-0002,
  ADR-0013) цена расширенной capability выше, чем выгода от вложенного bwrap;
* **Fallback-цепочки провайдеров** и **предварительная оценка токенов через
  tiktoken** — обе исключены из объёма решением от 2026-07-20 (спека блока
  A, раздел «Цель»). Fallback-цепочки требуют отдельной модели конфигурации
  мультипровайдерности вне текущего цикла; tiktoken-оценка даёт число, не
  совпадающее с токенайзером реального провайдера, и создаёт ложную точность
  там, где `prompt_tokens`/`cached_tokens` из самого ответа API уже дают
  точные цифры постфактум (§3.1 выше, ADR-0015 §6.3).

## 5. Концепции перенять (обновления наших документов)

1. **Auxiliary model** в конфигурации (`models.auxiliary`) — для curator, компакции, verifier-судьи. Дешевая модель для служебных LLM-задач. → ТЗ §13, §6.11.
2. **Режим автономии фиксируется при старте run** и не перечитывается из env/конфига во время исполнения — иначе prompt injection может эскалировать до yolo. → ТЗ §12, ADR-0010.
3. **Curator-инварианты Hermes** → ADR-0009/§18.1: только agent-created скиллы, pinned-флаг, защита скиллов из scheduled-задач, sidecar/SQLite-телеметрия (у нас уже SQLite trace — наш вариант сильнее), якорь created_at.
4. **Слой 1 curator применяется автоматически** (без proposals): переходы active→stale→archived обратимы и касаются только agent-created скиллов — гонять их через governance flow избыточно и противоречит yolo-first. Consolidation (слой 2) — по-прежнему только через proposals. → правка ADR-0009.
5. **Совместимость SKILL.md с agentskills.io** — бесплатная экосистема готовых скиллов. → ТЗ §7.
6. **Plan mode** (из HKUDS) — профиль «построй план, не исполняй» как четвертый режим рядом с supervised/auto/yolo. → ТЗ §3.6 (пост-MVP).
7. **Лимит байт на memory entrypoint** (из HKUDS memdir) — защита контекста от разрастания памяти. → §6.3/6.7.

## 6. Что мы делаем лучше (осознанные отличия — не менять)

* **Git-native память** с историей, rollback и single-writer: у Hermes память — файлы + sidecar JSON без версионирования, у HKUDS — memdir без Git. Наше главное отличие.
* **Sandbox по умолчанию + enforcement-first** (ADR-0002): у Hermes по умолчанию local-исполнение, безопасность — паттерны+approval; docker — опция. У нас гарантии не зависят от распознавания команд.
* **Typed critical-набор + Policy Engine как отдельный слой**: у обоих референсов политика размазана по коду.
* **Resumable runs как state machine с write-ahead checkpoint** (ADR-0005): у Hermes есть checkpoint_manager, но run не переживает процесс как первоклассная сущность.
* **Полный SQLite-trace каждого действия** и secret scan перед каждым коммитом (у референсов отсутствует — у них нет git-flows как концепции).
* **Skill governance через git proposals** для содержательных изменений скиллов.

## 7. Оценка трудозатрат, которые экономим

Адаптация environments (докер+local) ≈ экономит issue 10 почти целиком; DANGEROUS_PATTERNS ≈ половина issue 11-эвристик; frontmatter-парсер ≈ треть issue 14; дизайн компакции и curator — снимает проектные риски M5. Суммарно — недели работы и, что важнее, чужие «шишки» уже учтены (cron-referenced skills, YOLO-freeze, sidecar-телеметрия, protected head/tail при компакции).
