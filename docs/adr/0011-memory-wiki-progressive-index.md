# ADR-0011: Память как LLM-wiki — детерминированный индекс и прогрессивная загрузка

## Статус

Принято (реализация — после M5; layer «lint» и «sources» — второй эшелон)

## Контекст

Память агента (`memory/`, ADR-0004) сейчас плоская: `user/profile.md`,
`projects/<имя>.md`, `decisions/<тема>.md`. В контекст она попадает
целиком — `reader.py` конкатенирует все `memory/**/*.md` и **режет хвост**
по лимиту 16 КБ. Два следствия деградируют по мере роста:

* с ростом числа проектов целые файлы **молча выпадают** из контекста
  (усечение хвоста), причём приоритет фиксирован (`user > projects > decisions`),
  так что первым исчезает самое старое из `decisions/`, затем проекты;
* у агента **нет инструмента читать память по ходу run** — единственная
  точка чтения это инъекция на старте; `RememberTool` умеет только писать.
  То есть выбрать «загрузить нужную страницу проекта, когда она понадобилась»
  агент физически не может.

Референс — паттерн «LLM-wiki» (Karpathy): LLM инкрементально ведёт
persistent-wiki с навигационными файлами `index.md` (каталог) и `log.md`
(хронология), а «скучный bookkeeping» (перекрёстные ссылки, консистентность
каталога) автоматизируется. Прогрессивная загрузка через индекс — тот же
приём, что уже применён к скиллам (progressive disclosure §3.4, `read_skill`).

## Решение

Память эволюционирует в wiki с **детерминированным (код, не модель)** индексом
и **прогрессивной загрузкой** страниц по требованию.

### Раскладка

```
memory/
  index.md              # АВТОГЕН: каталог по категориям — в контекст
  log.md                # АВТОГЕН: append-only хронология изменений
  user/profile.md       # горячее, всегда в контексте
  projects/<slug>/
    overview.md         # frontmatter + описание (единственная обязательная страница проекта)
    notes.md            # свободные заметки (опц.)
    decisions.md        # решения по проекту (опц.)
  decisions/<тема>.md   # кросс-проектные решения (как сейчас)
  sources/<slug>/…      # raw-слой: неизменяемые исходники (спеки, доки)
```

### Контракт страницы проекта (модель обязана его знать)

`projects/<slug>/overview.md` начинается с YAML-frontmatter:

```yaml
---
name: AnimateYou
slug: animateyou
summary: бот генерации медиа по запросу; агрегатор API + агент
status: active          # active | paused | archived
tags: [bot, media, ai]
created: 2026-07-10
updated: 2026-07-10
---
```

Обязательные поля: `name`, `slug`, `summary`, `status`. `slug` обязан
совпадать с именем папки `projects/<slug>/` — иначе ошибка tool. `apply.py`
валидирует поля при заявке на `projects/<slug>/overview.md` — отсутствие
возвращается модели сразу как ошибка tool (тот же механизм ранней валидации,
что уже есть в `memory_tools.py`). Парсер frontmatter переиспользуется из
`skills/frontmatter.py` (`split_frontmatter`), вынесенного в нейтральный
модуль. Схема контракта дублируется в системном промте (`_MEMORY_GUIDE`),
включая точный шаблон frontmatter.

**Даты `created`/`updated` ведёт код, не модель.** При создании `overview.md`
проставляются `created` и `updated` = сегодня; при любом последующем
изменении страницы `updated` перештампуется. Код парсит frontmatter,
обновляет только даты и пересобирает блок, сохраняя тело нетронутым. Модель
эти поля может опускать — так они не устаревают (тот же принцип, что с
индексом: bookkeeping — на коде).

### Индекс и лог — детерминированно, кодом

`index.md` и `log.md` помечены `<!-- AUTOGENERATED -->` и **не редактируются
моделью** (правило в промте). Их ведёт единственный memory-writer в `_drain`,
под тем же writer-локом (ADR-0007), после применения заявок:

* `index.md` **перегенерируется** из frontmatter всех `projects/*/overview.md`
  (плюс перечисление `decisions/*` и `user/profile.md`). Секции:
  `## Проекты` (active/paused, сортировка по `updated` desc),
  `## Проекты (архив)` (archived — отдельно), `## Решения`, `## Пользователь`.
  Строка проекта: `- [name](path) — summary _(status · updated)_`; заголовок
  и краткое для `decisions/*` берутся из первого H1 и первого абзаца;
* в `log.md` **дописывается** строка на каждую применённую **контентную**
  заявку с парсимым префиксом: `## [YYYY-MM-DD] <op> | <path> | run <id8>`
  (дата — `applied_at`, UTC). Сам автоген (`index.md`/`log.md`) в лог не
  пишется. Ротация лога — будущий долг.

Ключевой инвариант: writer единственный и сериализован, поэтому регенерация
индекса из **полного текущего состояния** всегда консистентна — проблемы
last-writer-wins для каталога не возникает (в отличие от варианта, где индекс
правит модель через конкурентные `append`/`replace_section`).

### Прогрессивная загрузка

* `reader.py` инъектит в контекст **только `index.md` + `user/profile.md`**
  (обе — под лимитом), а не весь дамп. Устраняет молчаливое усечение.
  `decisions/*` и страницы проектов в контекст **не** попадают — они видны в
  индексе и подгружаются по требованию через `read_memory`. Нет `index.md`
  (свежая установка) → инъектится только `user/profile.md`.
* Новый tool **`read_memory(path)`** (read-only, LOW, аналог `read_skill`):
  агент по индексу подтягивает страницу проекта по ходу run. Регистрируется
  в `orchestrator.py` рядом с `RememberTool`.

### Lint — memory-curator (второй эшелон)

Команда `svarog memory curate` — аналог слоя 1 skill-curator'а (ADR-0009), но
**структурный**: у страниц памяти нет usage-телеметрии, поэтому вместо
lifecycle-по-usage — детерминированный аудит. Реализован первый срез
(report-only, без LLM и мутаций): ищет `orphan` (папка проекта без
`overview.md`), `invalid` (`overview.md` не проходит контракт), `stale`
(`status: active`, но `updated` старше `curator.stale_after_days`), `empty`
(пустые `.md`; `index.md`/`log.md`-автоген исключены). Печатает находки и
пишет отчёт в `artifacts/`; run'ы не блокирует, память не мутирует.

Отложено (нужна отдельная подсистема): семантический слой на LLM
(противоречия между страницами, дубли проектов) и материализация правок как
**memory-proposals** на ревью человеку (Flow B, ADR-0003) — для памяти
подсистемы proposals пока нет (в отличие от скиллов), это её собственный
дизайн-шаг.

### Raw-слой `sources/` (второй эшелон)

Исходники, которые агент «переварил» (внешние спеки, доки), кладутся
неизменяемыми в `sources/<slug>/` и **цитируются** из страницы проекта, а не
переписываются в память. Отражает трёхслойность Karpathy (raw → wiki →
schema); для Svarog schema-слой — это `_MEMORY_GUIDE` в промте. Реализовано:
`_MEMORY_GUIDE` документирует правило; `render_index` выводит их разделом
`## Источники` (навигация); `remember` гейтит неизменяемость — под `sources/`
разрешены только `create`/`delete`, `append`/`replace_section` отклоняются.
Читаются по требованию через `read_memory` (как любая страница).

### Порядок реализации

Первый срез (шаги 1–4) — самодостаточен: чинит усечение контекста и даёт
прогрессивную загрузку. Шаги 5–6 — второй эшелон. Миграция (7) — вне срезов,
делается отдельно позже.

1. Контракт: frontmatter (+ вынос `split_frontmatter` в общий модуль),
   код-управление датами, правила в `_MEMORY_GUIDE`.
2. `apply`/`writer`: валидация frontmatter + автоген `index.md` и `log.md` +
   штамповка `created`/`updated`.
3. `reader`: только `index.md` + `user/profile.md`.
4. tool `read_memory` + регистрация.
5. `svarog memory curate` (lint).
6. `sources/` raw-слой + гайд по ingest.
7. **(позже, отдельно)** Разовая миграция проектов из `user/profile.md` →
   `projects/<slug>/overview.md`. До неё `## Проекты` в индексе пуст и
   наполняется новыми проектами по мере работы; факты о работе/расписании
   (Северсталь и т. п.) остаются в `user/profile.md` как профиль, не проект.

## Последствия

* Контекст перестаёт раздуваться и молча терять файлы: на входе — компактный
  индекс, тяжёлое подгружается по требованию. Цена — лишний round-trip
  (`read_memory`), когда нужна конкретная страница.
* Индекс и лог не могут «забыться» моделью (тот же класс бага, что правило
  «результат — в файл»): их пишет код. Модель отвечает только за контент
  страниц и корректный frontmatter.
* Появляется зависимость от парсинга YAML-frontmatter в `apply`/`reader`
  (добавляется парсер frontmatter; сейчас его нет).
* `log.md` частично дублирует git-историю memory-репо (trailer `Run-Id`) и
  SQLite-трейс. Оставлен осознанно как человекочитаемый навигационный файл
  паттерна; пишется кодом, стоит дёшево.
* Правило `replace_section` (ADR-0004) для страниц проекта работает по
  заголовкам разделов внутри `overview.md`; frontmatter правится через
  `create`/regenerate, а не `replace_section`.
* Миграция отложена и не входит в первый срез: до неё раздел `## Проекты` в
  индексе пуст, а существующие проекты остаются описаны в `user/profile.md`.
  Это осознанно принято — новые проекты сразу пишутся в новую раскладку, а
  ретроспективный перенос AnimateYou/роплей-бэкенда делается позже (шаг 7).
* `created`/`updated` управляются кодом, поэтому `apply` при изменении
  `overview.md` пересобирает frontmatter-блок (парс → правка дат →
  сериализация), сохраняя тело. Это единственное место, где код правит
  контент страницы проекта; `replace_section` от модели трогает только
  разделы тела, не frontmatter.
