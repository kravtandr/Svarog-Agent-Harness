# ADR-0005: Resumable runs — базовая модель исполнения

## Статус

Принято

## Контекст

Approval через Telegram может занять часы — run обязан уметь ждать без удержания процесса и sandbox. Refuel требует переноса задачи между сессиями. Падение процесса не должно терять прогресс. Это три проявления одной потребности: run должен уметь останавливаться и продолжаться из сохраненного состояния. Проектировать loop как линейный вызов, а resumability добавлять потом — гарантированная переделка ядра.

## Решение

Agent run — state machine:

```text
pending → running → completed | failed | cancelled
running → waiting_approval → running
running → suspended → running
```

**Checkpoint** сохраняется в SQLite после каждой итерации loop и содержит: message history (или ссылку на нее), счетчики итераций/токенов/стоимости, незавершенный tool call (если есть), ссылки на workspace и branch.

Триггеры приостановки — единый механизм, разные причины:

| Триггер | Состояние | Возобновление |
|---|---|---|
| require_approval | `waiting_approval` | решение из любого интерфейса |
| порог refuel (итерации/токены) | `suspended` + `task_state.md` в Git | новая сессия пересобирает контекст |
| лимит стоимости/токенов | `suspended` | явное решение пользователя |
| падение/рестарт процесса | `suspended` (при старте: recovery незавершенных runs) | автоматически или по команде |

Разница refuel и обычного resume: обычный resume восстанавливает контекст дословно из checkpoint; refuel **пересобирает** контекст заново из `task_state.md` + Git (это и есть смысл refuel — сброс раздутого контекста).

Sandbox при приостановке останавливается (контейнер удаляется), workspace на диске сохраняется; при возобновлении создается новый контейнер над тем же workspace. Замораживание контейнеров (docker pause) — оптимизация, не базовый механизм.

## Последствия

* Loop с первого дня пишется вокруг checkpoint'ов — немного больше кода в MVP, но исключает переделку ядра.
* Approval становится асинхронным по построению: никакого «CLI ждет с открытым stdin» как основы механизма.
* Recovery после падения — просто частный случай resume.
* Идемпотентность tool calls важна на границе «tool вызван, результат не записан» — checkpoint фиксирует tool call до исполнения (write-ahead).
* **Refuel реализован через тот же suspend/resume** (`loop._refuel_suspend` + `_rebuild_after_refuel`): при достижении порога run уходит в `suspended` со сбросом контекста в `task_state.md` (раздутая история в checkpoint не сохраняется), процесс и sandbox освобождаются, а `resume` пересобирает контекст с нуля и создаёт новый sandbox над тем же workspace. Одноразовый CLI-`run` поднимается командой `svarog resume`; в долгоживущих процессах (`serve`/`telegram`) refuel-supervisor `GatewayService` (секция `supervisor`) сам поднимает refuel-suspended run'ы — refuel-приостановку отличаем по `refuel_pending` в последнем checkpoint'е, поэтому budget/max/approval-остановки супервизор не трогает. Естественный потолок числа авто-возобновлений — `max_iterations` (total не сбрасывается), поверх него `supervisor.max_auto_resumes`. `refuel_after_iterations > max_iterations` отключает refuel для коротких интерактивных задач.

## Автопродолжение после refuel (блок B, 2026-07-21)

Раздел дополняет решение выше и **заменяет** два утверждения из него:
формулировку «естественный потолок числа авто-возобновлений — `max_iterations`
(total не сбрасывается)» и подразумеваемое требование ручного `svarog resume`
для одноразового CLI-`run`.

### Что изменилось

* **`max_iterations` — лимит сегмента, а не всего run'а.** Условие цикла
  опирается на `LoopState.iterations_since_refuel`, который обнуляется при
  каждом сбросе контекста. Поле `state.iterations` остаётся тотальным
  счётчиком для отчётности и trace, но стоп-краном быть перестало.
* **Run продолжает себя сам.** При достижении порога refuel цикл пишет
  `task_state.md`, пересобирает контекст (`_autocontinue` →
  `_rebuild_after_refuel`) и продолжает работу, не выходя в `suspended`.
  Ручной `svarog resume` для долгой задачи больше не нужен — ни в CLI, ни в
  долгоживущих процессах.
* **Общие стоп-краны:** потолок раундов `runtime.max_refuel_rounds`
  (по умолчанию 12), бюджет токенов, бюджет стоимости. Все три проверяются на
  границе итерации.
* `max_refuel_rounds=0` возвращает прежнее поведение — приостановку на первом
  же refuel.

### Механизм выбирается конфигом, а не сосуществует

До блока B авто-продолжение существовало только в долгоживущих процессах — как
refuel-supervisor `GatewayService`. Теперь механизмов формально два, и они
разведены явно: при `max_refuel_rounds > 0` супервизор **не** поднимает
refuel-suspended run'ы (`supervise_once` возвращает пустой список).

Причина: при включённом автопродолжении refuel-приостановка означает
исчерпанный потолок, то есть намеренную остановку, решение по которой
принимает человек. Поднимать её супервизором значило бы отменять потолок —
и, поскольку ручной resume выдаёт новый бюджет раундов (см. ниже), давало бы
петлю, ограниченную только `supervisor.max_auto_resumes`.

Восстановление после падения процесса этим не затрагивается: оно идёт другим
путём — `recover_interrupted_runs` по протухшему heartbeat (ADR-0015 §0.5).

### Бюджет раундов

`LoopState.refuel_rounds` сериализуется в checkpoint, иначе падение процесса
обнуляло бы потолок само собой; старые checkpoint'ы читаются со значением `0`.

**Ручной `svarog resume` обнуляет счётчик:** явное действие человека выдаёт
новый бюджет раундов. Без обнуления resume после исчерпания потолка
немедленно упирался бы в него снова и был бы бесполезен.

### Durability и наблюдаемость

Порядок операций не изменился: `task_state.md` пишется и коммитится **до**
сброса истории, поэтому падение между записью и продолжением не теряет
прогресс. Наблюдаемость: `Run.meta["refuel_rounds"]`, сообщения пересобранного
контекста в trace на каждом раунде, уведомление через хук `on_notify`.

Файлы: `config/schema.py` (`max_refuel_rounds`), `runtime/checkpoint.py`
(`refuel_rounds`), `runtime/loop.py` (`_write_task_state`, `_autocontinue`,
`_refuel_suspend`, `resume`), `gateway/service.py` (`supervise_once`).
Reproducer'ы: `tests/test_loop.py`, `tests/test_refuel.py`,
`tests/test_resume.py`, `tests/test_gateway.py`.
