# ADR-0010: Yolo-first автономия — approval только в крайних случаях

## Статус

Принято

## Контекст

Основной сценарий использования — автономный запуск (`--yolo`): агент работает без наблюдения, подтверждения человека допустимы только в крайних, действительно важных случаях. Первоначальная модель «approval для high и critical» при таком сценарии превращается в шум: run постоянно висит в `waiting_approval` на обратимых действиях (push в task-ветку, удаление файла в workspace), и ценность автономного агента теряется. При этом существует класс необратимых/внешних действий, где ошибка не откатывается — там approval обязателен всегда.

## Решение

### Режимы автономии

| Профиль | high-действия | critical-набор |
|---|---|---|
| `supervised` | require_approval | require_approval |
| `auto` | notify | require_approval |
| `yolo` (по умолчанию) | notify | require_approval |

`notify` — новое policy-решение: действие выполняется сразу, пользователю уходит асинхронное уведомление, действие выделяется в trace. MVP-канал уведомлений — вывод CLI и отметка в trace; позже — Telegram/Web push.

### Неотключаемый critical-набор

Approval в любом режиме, конфигурацией не отключается: продовый деплой и изменение работающих сервисов; выдача секретов, платежи; необратимое удаление вне workspace; force-push/переписывание истории общих веток; merge/push в защищенные ветки; ослабление политик безопасности (сеть sandbox, secret scan).

Ключевое ограничение: critical-набор определяется **только по типизированным операциям** (deploy-tools, SecretStore, gitflow-компонент, policy-изменения) — не по bash-эвристикам. Это согласовано с ADR-0002: bash физически не может выполнить critical-действие из sandbox без сети, секретов и credentials. Эвристики могут эскалировать bash-команду до notify, но никогда не участвуют в разрешении critical.

### На что опирается безопасность в yolo

1. **Enforcement** (ADR-0002): sandbox-инварианты не зависят от режима автономии.
2. **Обратимость**: все изменения — в ветках и коммитах; протолкнуть необратимое состояние (merge в main, force-push) без человека нельзя.
3. **Бюджеты**: max_iterations, max_cost, max_tokens — стоп-краны, не требующие человека в цикле.
4. **Post-hoc review**: полный trace с выделенными notify-событиями заменяет pre-approval для обратимых действий.

## Последствия

* UX-акцент смещается с pre-approval на качество trace и уведомлений: notify-события должны быть заметными и легко ревьюиться пачкой.
* Механизм `waiting_approval` (ADR-0005) остается без изменений — просто срабатывает редко.
* Policy Engine получает четвертое решение `notify` и конфигурацию профилей (`policies.profiles` в svarog.yaml) + `protected_branches`.
* Ошибочная классификация high-действия в yolo означает его выполнение — поэтому все, что действительно должно останавливать агента, обязано попасть в типизированный critical-набор, а не полагаться на эвристики.
* Режим автономии и policy-конфигурация **фиксируются при старте run** и не перечитываются из окружения/файлов во время исполнения: иначе prompt injection мог бы эскалировать режим изнутри run (прием заимствован из hermes-agent, где YOLO-флаг замораживается при импорте процесса).
