# ADR-0009: Skill Curator — двухслойное кураторство библиотеки скиллов

## Статус

Принято (реализация — пост-MVP, после skill governance)

## Контекст

Self-improvement через skill proposals означает, что библиотека скиллов постоянно растет. Без кураторства она деградирует: дубликаты, устаревшие и сломанные скиллы, размытые описания. Деградируют и skill cards — а именно по ним модель выбирает навык, то есть качество карточек напрямую влияет на качество работы агента. Референс — curator в Hermes agent: разделение на дешевый механический слой и семантический LLM-слой.

## Решение

Curator — отдельный компонент с двумя слоями; полное описание поведения — §18.1 ТЗ.

### Слой 1: механический pruning (без LLM)

Детерминированные проверки по журналу использования (из trace) и метаданным: неиспользуемые скиллы, высокая доля неудачных применений, невалидная metadata, битые скрипты, ссылки на несуществующие tools. Обратимые lifecycle-переходы применяет сам; остальное — кандидаты для слоя 2. Дешевый, запускается часто. Usage-телеметрия хранится в SQLite (наш trace), а не во frontmatter SKILL.md — тот же принцип «операционные данные вне человекочитаемого контента», что и sidecar `.usage.json` в hermes.

### Слой 2: семантическая консолидация (LLM)

Работает по кандидатам слоя 1 и по библиотеке целиком: поиск дубликатов (embedding similarity при наличии VectorBackend, иначе LLM-анализ пар), предложения merge/extract/archive, улучшение `description` в skill cards. Дорогой, запускается по расписанию / после N новых скиллов / вручную (`svarog skills curate`).

### Инварианты (уточнены по анализу hermes-agent, см. docs/reference-analysis.md)

* curator работает только с **agent-created** скиллами (provenance-маркировка при создании); официальные и рукописные — вне зоны действия;
* **слой 1 применяет обратимые lifecycle-переходы автоматически** (`active → stale → archived`, реактивация) — без proposals: они не удаляют данные и откатываются одним действием; гонять их через governance противоречит yolo-first (ADR-0010);
* **содержательные изменения — только через proposals** (Flow B, ADR-0003): consolidation, merge, правки текста;
* никогда не удаляет — только архив (обратим); `pinned` выводит скилл из-под авто-переходов; скиллы, на которые ссылаются scheduled-задачи, не архивируются; якорь `created_at` защищает новые скиллы;
* слой 2 работает на auxiliary-модели и выключен по умолчанию (opt-in);
* consolidation не теряет информацию: объединяемые скиллы архивируются, не удаляются;
* каждый прогон оставляет отчет в trace и `artifacts/`;
* curator сам реализуется как official skill (`skill-curator`) поверх runtime-поддержки usage-статистики — это dogfooding платформы.

## Последствия

* Runtime уже в MVP должен логировать skill usage в trace (SkillLoad + исход run'а) — иначе слою 1 не с чем работать; сама автоматизация curator подключается позже.
* Слой 2 без vector DB деградирует до LLM-сравнения пар — работает, но дороже; с Qdrant — дешевле.
* Порог «скилл не использовался N дней» и расписание — конфигурация, не константы.
