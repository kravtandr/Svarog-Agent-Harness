# ADR-0004: Конкурентный доступ к Git-памяти — single writer

## Статус

Принято

## Контекст

ТЗ обещает одновременную работу через CLI, Telegram и Web. Два параллельных run'а, пишущих в `memory/`, создают merge-конфликты в markdown-файлах, которые некому разрешать: автоматический merge markdown ненадежен, а звать человека на каждый конфликт памяти — неприемлемый UX.

## Решение

Run'ы никогда не пишут в memory-репозиторий напрямую. Вместо этого:

1. Run формирует **MemoryChangeRequest** — структурированный объект: файл, операция (create / append / replace-section / delete), содержимое, source run_id.
2. Запросы попадают в очередь (MVP: таблица SQLite; server-режим: pluggable queue backend, ADR-0007).
3. Единственный **memory writer** применяет запросы строго последовательно, коммитит каждый примененный запрос отдельным коммитом с trailer `Run-Id:`.
4. Конфликты разрешаются last-writer-wins; проигравшая версия не теряется — она остается в Git-истории.
5. Push в remote (если настроен) — фоновый, батчами.

MVP-реализация: writer — asyncio-задача в том же процессе, сериализация через очередь в SQLite (переживает рестарт процесса).

## Последствия

* Память становится слегка асинхронной (eventual): изменение видно другим run'ам после применения из очереди. Для памяти это приемлемо.
* Чтение памяти — без ограничений, напрямую из working tree.
* Формат replace-section требует стабильных якорей в markdown (заголовки секций) — учитывается в конвенциях файлов памяти.
* Один и тот же механизм работает для одного процесса (MVP) и для нескольких (server) — меняется только backend очереди.
