# ADR-0020: Memory-proposals и Dream — семантический слой памяти под ревью

## Статус

Принят и реализован (блок C, 2026-07-23). Объём цикла — подсистема
memory-proposals и Dream поверх неё. Вне цикла: ревью памяти из чата и
Telegram, ротация `log.md`, семантический слой skill-curator'а (ADR-0009 слой
2 — отдельная сущность).

## Контекст

Память умела расти, но не умела приходить в порядок. `memory/curator.py` даёт
детерминированный аудит — находит `orphan`, `invalid`, `stale`, `empty`, — но
ничего не чинит, и в его докстринге прямо назван блокер: «подсистемы
memory-proposals пока нет».

ADR-0011 зафиксировал эту подсистему как отложенный дизайн-шаг: «материализация
правок как memory-proposals на ревью человеку (Flow B, ADR-0003) — для памяти
подсистемы proposals пока нет (в отличие от скиллов), это её собственный
дизайн-шаг». Без неё не мог появиться и семантический слой: механизм, который
сам замечает противоречия и дубли, обязан чем-то оформлять свои выводы.

Разбор HKUDS/nanobot называл этот механизм Dream — «вторая стадия памяти,
фоновая консолидация, отдельно от curator'а». Это был последний незакрытый
пункт разбора.

## Решение

### На ревью уходит только Dream — граница по автору, а не по операции

Заявки обычных run'ов (`remember`) применяются напрямую, как раньше. ADR-0003
прямо запрещает обратное: «branch+review на каждое обновление убивает идею
памяти». Flow A не изменился.

Запрет прямой записи для Dream реализован **структурно**: в профиле Dream
инструмент `remember` не регистрируется вовсе. Конфигурация, которую можно
перепутать, здесь не участвует.

### Proposal живёт в очереди, а не в git-ветке

Скопировать Flow B напрямую нельзя. `SkillRepoFlow.create_proposal` делает
`checkout` отдельной ветки в рабочем дереве, а в memory-репозитории параллельно
коммитит writer в текущую ветку. Ветвление под работающим writer'ом — гонка;
merge markdown-веток — ровно то, чего ADR-0004 избегал осознанно.

Поэтому proposal — строка в `memory_proposals` с пачкой
`MemoryChangeRequest` в поле `changes`. Память не тронута до решения человека.

**Одобрение** перекладывает заявки в штатную очередь writer'а: применение,
secret scan, коммит с trailer `Run-Id` и перегенерация `index.md` идут прежним
путём. Единственный писатель по ADR-0004 сохранён, merge-конфликтов markdown
нет по построению. **Отклонение** только меняет статус.

**Предпросмотр считается на текущем состоянии памяти**, а не замораживается при
создании: между предложением и одобрением память меняется, а `replace_section`
ищет секцию по якорю-заголовку и `update_field` правит одно поле — обе операции
осмысленно переприменяются к изменившемуся файлу. Расхождение `memory_head` с
текущим HEAD показывается человеку предупреждением.

### Единица решения — один связный замысел

Dream за прогон создаёт N proposal'ов, а не один на всё. Замысел может трогать
несколько файлов («слить два проекта» — правка одного `overview.md` и архивация
другого), поэтому `changes` — список. Частичное одобрение получается даром, без
синтаксиса выбора вроде `--only 1,3,5`.

### Два ограничителя

**delete запрещён для непустых страниц.** Удаление содержательной страницы —
потеря, а не консолидация. Для «проект закончился» есть
`update_field status=archived`: страница остаётся, `index.md` уводит её в
раздел архива. Это перенос в память инварианта ADR-0009 «никогда не удаляет —
только архивирует, обратимо». Правило живёт в коде, а не в промте: модель не
должна иметь возможности обойти его собственной интерпретацией.

**Потолок непросмотренных предложений** (`dream.max_pending`, по умолчанию 20).
При достижении прогон завершается сразу, не потратив ни одного токена. Иначе
ежедневная джоба при неактивном человеке копит мусор без границы и платно.

### Dream — профиль run'а, а не особый исполнитель

`RunProfile.DREAM` регистрирует только `read_memory` и
`propose_memory_change`. Не регистрируются: файловые tools, `bash`,
`spawn_child_run`, MCP-tools, `read_skill`, `update_plan`, `schedule.create`,
`create_skill_proposal` и `remember`.

Причина шире, чем «Dream не пишет в память». Dream — единственный run, который
запускается без человека **и** обрабатывает содержимое, попавшее в память из
внешних источников (`sources/`, заметки прошлых run'ов). Дать такому run'у
shell и файловые tools означало бы, что текст в памяти управляет исполнением.

Структурный аудит прогоняется **кодом до** модели, его находки подаются в текст
задачи как установленные факты. Перепроверять детерминированный результат
языковой моделью — трата токенов и источник разногласий. Модель делает то,
чего аудит не умеет: ищет дубли и противоречия по смыслу.

### Выключатель: конфиг — гейт на заводке, дальше джоба

`system:memory-dream` заводится в `ensure_system_jobs` только при
`dream.enabled: true`. По умолчанию `false` — как `curator.semantic` по
ADR-0009: механизм опинионейтед и стоит денег.

После заводки управление переходит к джобе: `svarog cron enable|disable`.
Конфиг не переопределяет решение человека на живой джобе — это сохраняет
инвариант блока D, по которому код не возвращает включённой джобу, которую
человек выключил.

Следствие, названное вслух: `dream.enabled: false` после заводки не удаляет и
не выключает уже созданную джобу. Конфиг заводит, `cron disable` останавливает.

## Последствия

* Отложенный шаг ADR-0011 закрыт: у памяти появился семантический слой, а у
  структурного аудита — потребитель его находок.
* Память впервые может меняться по инициативе механизма, а не человека или
  run'а, — но только через явное одобрение, и Flow A при этом не изменился.
* Системных джоб стало две; `run_job` в CLI получил диспетчеризацию по имени
  джобы, а `JobRunRequest` — поле `name`.
* Одобренный proposal применяется не мгновенно, а при следующем сливе очереди
  (`svarog memory flush` или конец следующего run'а). Память eventual по
  ADR-0004 — это её обычное свойство, а не новое ограничение.
* Proposal, пролежавший долго, может стать неприменимым: его заявка честно
  падает в `failed` в очереди writer'а, не задевая соседние.
