# ADR-0006: Секреты инжектируются на execution-слое

## Статус

Принято

## Контекст

ТЗ требует «секреты не доступны автоматически», но агенту для реальной работы нужны git-credentials, токены MCP-серверов, ключи внешних API. Нужен механизм выдачи, при котором значения секретов не попадают в контекст LLM (иначе они утекают в trace, логи и потенциально в ответы модели).

## Решение

* **SecretStore** — pluggable интерфейс. MVP-backend: **JSON-файл `~/.svarog/secrets.json` с правами `0600` (содержимое в открытом виде, без шифрования)** плюс fallback на env-переменные (`LayeredSecretStore`: файл → env). Защита MVP-хранилища держится на правах доступа ФС и на том, что файл в denylist/`.gitignore`, а не на шифровании содержимого. Шифрование файла (Fernet/KMS), HashiCorp Vault, corporate KMS — следующие backend'ы за тем же интерфейсом.
* Агент оперирует только **именами** секретов (`secrets: [github-token]` в конфигурации tool/skill), никогда значениями.
* Инжекция происходит на execution-слое, минуя LLM:
  * env-переменные sandbox-контейнера (только явно выданные);
  * конфигурация MCP-коннектора при старте;
  * git credential helper host-компонента (для push).
* Выдача секрета tool/skill — policy-решение уровня `critical` (approval обязателен, решение можно сохранить как постоянное правило в `policies/`).
* **Redaction**: известные значения секретов вырезаются из tool outputs, trace и логов до записи.

### Защита Git-репозиториев (проект и репозитории публичны)

Svarog — open-source проект в открытом доступе, и agent-home/рабочие репозитории могут иметь публичные remotes. Каждый коммит считается потенциально публичным:

* **pre-commit secret scan** обязателен во всех трех git flows (ADR-0003): обнаружен секрет → commit блокируется, run получает ошибку с указанием файла;
* сканер: паттерны известных форматов ключей (AWS, GitHub tokens, private keys, …) + entropy-эвристика + точные значения из SecretStore;
* файлы секретов (`.env`, key-файлы, secret store) — в denylist путей для `write_file` и коммитов; `svarog init` генерирует соответствующий `.gitignore`;
* та же проверка повторяется перед push (вторая линия против коммитов, сделанных в обход агента);
* для репозиториев с публичным remote опция не отключается policy;
* секрет, попавший в историю, считается скомпрометированным: сначала ротация, затем очистка истории.

## Последствия

* LLM физически не видит значения секретов; утечка через контекст исключена по построению (кроме случая, когда команда сама печатает секрет — это ловит redaction).
* Каждому official skill в manifest указываются требуемые секреты — прозрачно для review.
* Redaction — best-effort (секрет может быть закодирован); основная гарантия — невыдача секрета без approval, redaction — вторая линия.
* MVP-хранилище (`FileSecretStore`) держит значения в открытом JSON — компрометация файла = компрометация секретов. Поэтому: файл строго вне репозитория, права `0600`, denylist-путь. Кто не готов к plaintext-файлу — использует `EnvSecretStore` (значения в окружении процесса) или дожидается шифрованного backend'а.
