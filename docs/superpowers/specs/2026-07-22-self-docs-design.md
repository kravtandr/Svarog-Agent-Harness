# Self-docs для external executor'ов

**Дата:** 2026-07-22 (пересмотрено 2026-07-23 по итогам симуляции)
**Статус:** реализовано
**Скоуп:** внешние executor'ы с MCP (claude-code, opencode). Нативный `AgentLoop` — вне скоупа.

## Проблема

Св_арог не умел отвечать на вопросы пользователя о себе самом — командах,
фичах, архитектурных решениях. При основном сценарии использования (внешний
executor в sandbox) агент не имеет доступа к документации проекта: она лежит на
хосте (`README.md`, `AGENTS.md`, `docs/adr/`), а контейнер видит только
`/workspace` (проект пользователя) и launch-файлы.

Задача: на вопрос «что такое refuel loop?» или «три git-flow сварога» агент
должен отвечать по реальной доке, а не по претрейну.

## Решение: reverse-tool `read_svarog_docs` через bridge

Документация отдаётся MCP-тулом на bridge — по образцу существующего
`read_skill` (`BridgeControl._build_tools`). Файлы в контейнер НЕ монтируются.

### Почему не файловый ro-mount (пересмотр первоначального дизайна)

Первая версия монтировала стейдж-копию доков `:ro` в `/opt/svarog-docs`.
Симуляция на opencode показала, что это принципиально нерабочий путь:

```
X = resolve($), Y = process.cwd();
if (!X.startsWith(Y + sep) && X !== Y)
  throw new S({message: `filePath resolves outside the working directory: ${$}`});
```

`read` OpenCode жёстко отвергает любой путь вне cwd. Агент вызвал
`read /opt/svarog-docs/INDEX.md` и получил отказ. Mount при этом был корректен
(подтверждено `docker inspect` и чтением под uid агента) — дело в самом
инструменте агента. Положить доки внутрь `/workspace` нельзя: их подберёт
git-flow (Flow C) и закоммитит в репозиторий пользователя.

Исходный дизайн опирался на утверждение «MCP есть только у claude-code» из
docstring `agents/__init__.py`. **Оно было устаревшим:** фактически
`opencode.py` даёт `mcp=True` (включено спайком 21.07.2026). Матрица в
docstring и в ADR-0016 §20 исправлена.

### Фактическая матрица

| Адаптер | mcp | Образ в репо | Self-docs |
|---|---|---|---|
| claude-code | ✓ | ✓ | ✓ |
| opencode | ✓ | ✓ | ✓ |
| codex | ✗ | нет | нет (указатель был бы ложью) |

MCP покрывает оба развёртываемых executor'а.

## Компоненты

### `runtime/self_docs.py`

- `resolve_docs_root() -> Path | None` — корень репо по маркерам `README.md` +
  `docs/adr/`. None → фича мягко отключается.
- `build_docs_index(root=None) -> str` — каталог документов с заголовками ADR
  (парсятся из первой заголовочной строки).
- `read_doc(rel, root=None) -> str` — содержимое по виртуальному пути.
- `self_docs_hint(tool_name) -> str` — блок для контекст-файла агента.

**Пространство имён — allowlist, не путь:** `README.md`, `AGENTS.md`,
`adr/<файл>.md`. Произвольные пути не резолвятся, а отвергаются, поэтому обход
за корень невозможен по построению (не проверкой, а конструкцией).

### `tools/docs_tools.py` — `ReadSvarogDocsTool`

`read_svarog_docs`, `action_type="docs.read"`, `RiskLevel.LOW`, read-only.
Без `path` — каталог; с `path` — документ. Вывод обрезается на 40 000 символов
(ADR-0015 весит 60 КБ).

### Регистрация и проводка

- `BridgeControl._build_tools()` регистрирует tool при `self_docs=True` и
  найденном docs-root; флаг приходит из `executor.external.self_docs`
  (default `true`) через orchestrator.
- Адаптеры получают `context_files(memory, skill_cards, self_docs: bool)` и
  называют tool в своём неймспейсе: `mcp__svarog__read_svarog_docs` /
  `svarog_read_svarog_docs`. У codex ветка отсутствует — `mcp=False`.

## Сопутствующий баг: контекст не доходил до OpenCode

Симуляция вскрыла дефект **шире self-docs**. Св_арог писал
`~/.config/opencode/AGENTS.md`, полагаясь на автоподхват; в opencode 1.18.4
этот файл в контекст модели не попадает.

Проверено канарейкой (уникальный маркер в памяти → печатается в `AGENTS.md`):

| | Итераций | Как ответил |
|---|---|---|
| До | 2 | понадобился вызов `svarog_read_memory` |
| После | 0 | назвал прямо из контекста |

Молча терялись правила памяти, гайд `ask_user`, правила работы с файлами и
карточки скиллов. Фикс — явный ключ `instructions` в `opencode.jsonc`
(`provider_files`), указывающий на файл контекста в state volume.

## Наблюдения по поведению модели

Замеры на `gpt-oss-120b` + opencode (изолированный стенд, N=5):

| Изменение | Вызовов MCP-тула |
|---|---|
| Исходное описание тула | 1 из 3 |
| Описание с «документации НЕТ в workspace» | 5 из 5 |

Модель также брала идентификатор из каталога и звала нативный
`read /workspace/<путь>` — каталог теперь явно предупреждает, что это
идентификаторы, а не файлы.

**Остаточное ограничение.** Надёжность зависит от instruction-following модели
и делится по классу вопроса:

- «перечисли ADR» (модель знает, что не знает) → 5/5 вызовов;
- «три git-flow сварога» (модель уверена, что знает) → на `gpt-oss-120b` чаще
  0 вызовов, ответ из претрейна;
- тот же вопрос на claude-code → каталог → ADR-0003 → верный ответ.

Это предел конкретной модели, а не дефект харнесса. Дальнейшая подкрутка
промпта даёт падающую отдачу.

## Границы (non-goals)

- Нативный `AgentLoop` не покрывается.
- Не отдаются `TASK.md` (77 КБ ТЗ), `docs/superpowers/`, `simulation/`.
- Нет индексации/эмбеддингов/RAG и нет поиска — только каталог и чтение.
- Файловый mount доков снят полностью.
