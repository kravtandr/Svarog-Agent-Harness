# Блок D: планировщик как источник запуска run'ов

## Контекст

Разбор [HKUDS/nanobot](https://github.com/HKUDS/nanobot) (MIT) выделил блок D:
запуск run'ов по расписанию, heartbeat-файл как источник задач и durable-очередь
внешних триггеров. Перед началом работ проведена инвентаризация кода — после
того, как в блоке B выяснилось, что объявленный отсутствующим механизм в
проекте уже был.

**Инвентаризация подтвердила: пользовательского планировщика нет.** Ни cron, ни
расписаний, ни отложенных задач. При этом найдены механизмы, на которые
планировщик обязан опираться, а не дублировать их:

* **цикл супервизора** (`gateway/service.py:786`) — рабочий образец
  «бесконечный цикл с интервалом», с троттлингом отдельных задач внутри тика
  (`_GC_INTERVAL_SEC = 3600`);
* **durable-очередь в SQLite с межпроцессным локом** (`memory/writer.py`,
  `storage/locks.py`) — образец надёжной доставки, уже работающий в проекте;
* **per-workspace lease** (`Run.workspace` + `Run.heartbeat_at`,
  `acquire_workspace_lease`) — готовый механизм, отклоняющий второй живой run
  на том же рабочем дереве;
* **телеметрия загрузки скиллов** (`SkillLoad`, `storage/models.py:257`) —
  связь «скилл ↔ run», через которую закрывается точка расширения в
  `skills/curator/pruning.py:12-13,62`, где прямо написано: «scheduled-задач в
  текущем срезе нет — точка расширения помечена в коде»;
* **обещание в `TASK.md` §18.1**: куратор запускается «в моменты простоя агента
  по интервалу, после добавления N новых скиллов» — не реализовано, потому что
  запускать было нечем.

Инвентаризация также обнаружила расхождение документации с кодом, не связанное
с блоком D, но чинимое в том же цикле: `TASK.md:559` обещает Redis для очередей
и локов, тогда как фактическая реализация — SQLite и `fcntl.flock` (ADR-0007).

## Цель

Дать проекту источник запуска run'ов, отличный от человека, не размывая при
этом заморозку прав, на которой держится модель безопасности.

Объём этого цикла — **ядро расписания и системные джобы**. Вне объёма и решается
отдельными циклами: heartbeat-файл как источник задач; durable-очередь внешних
триггеров; полный cron-синтаксис (см. §2).

---

## 1. Хранилище

Новая таблица `cron_jobs` (SQLAlchemy-модель + alembic-миграция). Поля:

| Поле | Назначение |
|---|---|
| `id`, `name` | идентификатор и человекочитаемое имя |
| `schedule_kind`, `schedule_spec`, `tz` | вид расписания и его параметры (§2) |
| `task` | текст задачи, с которым создаётся run |
| `workspace`, `session_id` | куда исполнять и куда доставлять результат |
| `autonomy`, `config_digest` | **замороженные** права (§6) |
| `enabled`, `protected` | активность и защита системных джоб (§8) |
| `origin` | кто создал: `human` / `agent` / `system` |
| `next_run_at`, `last_run_at`, `last_status`, `run_count` | состояние и история |

Таблица, а не JSON-файл: состояние проекта и так в SQLite (ADR-0007);
мультиарендность получает изоляцию даром (у тенанта своя БД, ADR-0012);
атомарный захват `next_run_at` требует транзакции, которую на файле пришлось бы
изобретать заново.

## 2. Расписание без новой зависимости

Поддерживаются два вида:

* `every` — интервал (например, каждые 6 часов);
* `daily_at` — время суток `HH:MM` в указанной таймзоне.

Полный cron-синтаксис в этот цикл **не** входит: он требует зависимости
(`croniter` или собственного парсера), а перечисленного достаточно и для
системных джоб, и для типовых пользовательских. Это осознанное упрощение;
расширение до cron-выражений — отдельный шаг, если появится спрос.

Расчёт следующего срабатывания — чистая функция от текущего момента и
параметров расписания, без обращения к БД: это делает её тестируемой без
подмены времени в глобальном модуле.

## 3. Демон и защита от двойного срабатывания

Новая команда `svarog scheduler` — цикл с интервалом из конфига.

**`serve` джобы не исполняет.** Это документируется явно: планировщик — отдельный
процесс, и запуск сервера сам по себе расписание не оживляет.

Два экземпляра демона не должны дублировать запуск одной джобы. Два рубежа:

1. существующий межпроцессный файловый лок (`storage/locks.py`, тот же, что у
   писателя памяти) вокруг тика — снимает конкуренцию в типичном случае;
2. **атомарный захват** джобы: обновление `next_run_at` условием
   compare-and-set внутри транзакции — настоящий рубеж, работающий и при
   отказе первого.

## 4. Занятый workspace

Per-workspace lease уже отклоняет второй живой run на том же рабочем дереве.
Планировщик не считает это ошибкой: джоба, чей workspace занят интерактивной
работой, **пропускает тик** и пробует на следующем, не теряя расписания и не
мешая человеку. Факт пропуска попадает в `last_status`.

Приём заимствован у nanobot (`defer-until-idle`), но опирается на существующий
lease, а не на новый механизм.

## 5. Пропущенные срабатывания

Если демон не работал сутки, джоба срабатывает **один раз**, а `next_run_at`
двигается вперёд от текущего момента.

Контракт — **at-least-once без catch-up**: догоняющего шторма не будет. Это
записывается явно, потому что противоположное поведение (отработать все
пропущенные тики) — частое и разрушительное умолчание у самодельных
планировщиков.

## 6. Замороженные права

При создании джобы (человеком или системой) либо при её approval (для
агентских, §7) в джобу записываются режим автономии и дайджест
security-конфига — тот же `config_digest`, что снимается при старте run
(ADR-0015 §0.4).

При срабатывании дайджест сверяется с текущим. Расхождение — джоба
**отключается** с внятной причиной и ждёт человека; run не создаётся. Это тот
же fail-closed, что у `resume` при config drift, и то же правило: последующее
ослабление конфига не повышает прав уже одобренной джобы.

Агентская джоба наследует режим автономии создающего её run'а — выдать себе
больше, чем есть у текущего run'а, агент не может.

## 7. Создание агентом через approval

Новый инструмент `schedule_task` с типом операции `schedule.create`, который
добавляется в неотключаемый `CRITICAL_ACTIONS` (`policy/engine.py:29`).
Следствие по устройству `PolicyEngine`: approval требуется **в любом режиме
автономии, включая yolo**, и конфигурацией не отключается.

Обоснование — прямо из ADR-0010: critical-набор определяется по типизированным
операциям, и в него попадает всё, что «действительно должно останавливать
агента». Джоба переживает run, поэтому инъекция из файла без этого гейта
получила бы способ закрепиться: заставить агента запланировать повторяющуюся
работу, которая исполнится позже и с новым конфигом.

До подтверждения джоба существует в состоянии `enabled = false` и не
срабатывает. Отказ в approval оставляет её выключенной.

## 8. Системные джобы

Регистрируются кодом при старте демона, идемпотентно (повторный старт не
плодит дубликатов), с `origin = system` и `protected = true`. Агентский
инструмент не может их изменить или удалить.

**Первый потребитель — механический слой skill-куратора по интервалу.** Это
закрывает обещание `TASK.md` §18.1, невыполнимое до появления планировщика.
Dream из блока C встанет сюда же.

**Точка расширения в `pruning.py` закрывается через существующую телеметрию.**
Скилл считается используемым, если он загружался в run'ах включённой джобы:
связь `SkillLoad` → `Run` → `cron_job_id` уже есть, и определять её по тексту
задачи (что было бы хрупко) не требуется.

## 9. Наблюдаемость и CLI

* `Run.meta["cron_job_id"]` связывает run с породившей его джобой;
* в джобе — `last_run_at`, `last_status`, `run_count`;
* команды: `svarog cron add|list|show|enable|disable|remove|run-now`,
  `svarog scheduler`.

`run-now` исполняет джобу немедленно, не сдвигая расписание — для проверки
задачи без ожидания срабатывания.

## 10. Тестирование

Текущий момент передаётся параметром, а не читается из глобального модуля:
в тестах никаких `sleep` и никакой подмены `datetime`.

Покрывается:

* расчёт следующего срабатывания для обоих видов расписания, включая переход
  через полночь и таймзону;
* single-flight: два одновременных тика не порождают два run'а по одной джобе;
* занятый workspace — тик пропускается, расписание не теряется;
* отсутствие догоняющего шторма после долгого простоя демона;
* расхождение дайджеста конфига отключает джобу и не создаёт run;
* агентский `schedule_task` требует approval даже в yolo;
* системная джоба не удаляется агентским инструментом;
* скилл, использованный run'ом включённой джобы, не архивируется куратором.

Перед коммитом: `ruff check`, `ruff format`, `mypy`, `pytest` — зелёные.

## Границы блока D

Не входит и решается отдельными циклами: heartbeat-файл как источник задач;
durable-очередь внешних триггеров с доставкой at-least-once; полный
cron-синтаксис. Блок C (Dream) идёт следующим и использует системные джобы
этого блока как механизм периодичности.
