# AGENTS.md — инструкции для работы с репозиторием Svarog

## Что это за проект

**Svarog** — open-source Git-native runtime для self-hosted ИИ-агентов: скиллы, sandboxed execution, Git-память, refuel loops, approval policies, полный audit trace. Это платформа для сборки агентов, а не готовый агент и не workflow/граф-фреймворк.

Именование: репозиторий **Svarog-Agent-Harness**, PyPI-пакет `svarog-harness`, Python-модуль `svarog_harness`, CLI-команда `svarog` (§0 TASK.md).

Проект публичный. Любой коммит потенциально виден всем — секретов в репозитории быть не должно ни в каком виде.

Референсные проекты для изучения склонированы в `/home/kravtandr/reference/` (hermes-agent, hkuds-openharness); анализ — в `docs/reference-analysis.md`.

## Как работать

1. Перед задачей — прочитать соответствующий раздел `TASK.md` и связанные ADR; задачи брать в порядке `docs/first-issues.md`.
2. Работать в feature-ветке; в `main` — только рабочее состояние.
3. Перед коммитом: `ruff check` + `ruff format`, `mypy`, `pytest` — всё зелёное.
4. Интерфейсы (`ModelProvider`, `SandboxBackend`, `QueueBackend`, `SecretStore` и т.д.) закладывать сразу, даже при единственной реализации (ADR-0008).
5. Правила зависимостей из `repo-structure.md` не нарушать: `cli` → `runtime` → компоненты → `storage`/`trace`; никаких импортов из `cli` в ядро.

## Правила коммитов

* Формат — Conventional Commits: `feat|fix|docs|refactor|test|chore(scope): описание`; scope — модуль (`runtime`, `policy`, `sandbox`, `skills`, `memory`, `gitflow`, `cli`, …).
* Сообщения на английском, в императиве, ≤72 символа в заголовке; в теле — «почему», если неочевидно.
* Один коммит — одно логическое изменение. Никаких `wip`, `fixes`, `misc` в `main`.
* Если коммит закрывает задачу из backlog — указать её: `feat(policy): add approval flow (#13)`.
* **Секреты**: никаких токенов, ключей, `.env`, URL с credentials — ни в коде, ни в тестах, ни в фикстурах, ни в истории. В тестах — только заведомо фейковые значения. Секрет попал в историю → он скомпрометирован: ротация, затем очистка истории.
* Не коммитить сгенерированное и локальное: `__pycache__`, `.venv`, артефакты сборки, IDE-файлы.

## Правило синхронизации документации и кода

При любом изменении архитектуры — **в том же PR/наборе коммитов**:

1. **Обновить все связанные документы**: затронутые разделы `TASK.md`, ADR (устаревший ADR не редактировать задним числом — пометить `Статус: заменен ADR-XXXX` и написать новый), `repo-structure.md`, `first-issues.md`, этот файл.
2. **Удалить легаси немедленно**: старый код, потерявший смысл после изменения, удаляется в том же изменении — не помечается deprecated «на потом» (до публичных релизов у проекта нет обязательств обратной совместимости).
3. **Никакого мёртвого кода**: закомментированные блоки, неиспользуемые функции/импорты/зависимости, «оставим на всякий случай» — не проходят review. История хранится в Git, а не в комментариях.

Рассинхронизация документации и кода — это баг того же уровня, что и падающий тест.
